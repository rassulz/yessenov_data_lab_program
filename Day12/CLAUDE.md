### The ONE intentional change from the notebook (this is the whole point)
The notebook's image2vec uses **ResNet18**, which embeds *images only* — it cannot take
a text query. To search **by text**, swap it for **CLIP**, whose text encoder and image
encoder write into the *same* space, so a sentence vector can be compared to image
vectors by cosine. Everything downstream stays the notebook's.

## Data source — Flickr30k (Hugging Face), STREAMED (no full download)
**Dataset:** `nlphuji/flickr30k` — real everyday photos (people, animals, food, streets),
each with **5 human captions**. The full set is several GB, so we **stream** it and pull
only a small sample — we never download or cache the whole thing:

    from datasets import load_dataset
    # revision="refs/convert/parquet" = HF's auto-converted Parquet copy. Required because
    # datasets>=4 dropped loading scripts and this repo ships flickr30k.py. Same schema.
    stream = load_dataset("nlphuji/flickr30k", revision="refs/convert/parquet",
                          split="test", streaming=True)
    rows = list(stream.take(1500))     # only ~1.5k images fetched, not the full set

`streaming=True` fetches images over the network as we iterate and `.take(N)` stops
early, so only the ~1.5k rows we use come down (a few hundred MB, saved to `images/`).
Each row has `image` (a PIL image — use it directly, no file walking) and `caption`
(a list of 5 reference captions). Keep the **captions** array alongside the images —
they are the free evaluation set (see stretch goals).

Why this one: real, varied scenes make text queries actually hit ("people laughing",
"sunset over water", "a plate of food"), and the captions give a **built-in way to
measure accuracy**. Start with ~1.5k images (seconds to embed on the RTX 3050), then
raise `take(N)` to scale up.

## Build steps (each maps to a notebook tool)
1. **Load images** — stream Flickr30k and `.take(1500)` (see Data source; no full
   download). Each row's `image` is a PIL image (`.convert("RGB")` to be safe). **Save
   each one to `images/{i}.jpg`** and use that path as its label, so the Streamlit app
   can show it later (Tool 8 stores labels + embeddings, the app displays images by
   path). Keep the captions array too.
2. **Load CLIP** — `model, _, preprocess = open_clip.create_model_and_transforms(...)`,
   `tokenizer = open_clip.get_tokenizer("ViT-B-32")`, `model.eval().to(device)` using
   `pickDevice()`.
3. **Embed every image** — same batched loop as Tool 1, but
   `feat = model.encode_image(batch_of_preprocessed_imgs)`; concatenate into
   `img_vectors` (`N × 512`). Show a `tqdm` bar.
4. **Embed the query** — `tokens = tokenizer([query]); tvec = model.encode_text(tokens)`.
5. **Search** — reuse the notebook's
   `cosineNeighbors(tvec, img_vectors, paths, topn=5)` → top image paths + scores
   (it normalizes internally, so cosine == dot product).
6. **Inspect** — run 3–4 real queries ("people laughing", "sunset over water",
   "a plate of food") and eyeball the top-5; note one hit and one miss.

## Notes / gotchas
- **Normalize before cosine.** CLIP vectors are not unit length; the notebook's
  `cosineNeighbors` normalizes internally, so if you write your own dot-product path,
  L2-normalize both sides first.
- **Two encoders, one space.** Use CLIP's *own* `preprocess` for images and its *own*
  tokenizer for text — swapping in another resize/normalize breaks the alignment.
- **RGB + robustness.** `.convert("RGB")` (PNGs may be RGBA/greyscale); wrap
  `Image.open` in `try/except` so one corrupt file does not kill the run.
- **First run downloads weights** (~350–600 MB) once, then caches — be patient on run 1;
  wrap the load in `try/except` for offline.
- **Device / GPU.** Reuse `pickDevice()` — it picks the **NVIDIA GPU (CUDA)** on this
  laptop (RTX 3050) automatically. To actually use the GPU, install the **CUDA build** of
  torch (`pip install torch --index-url https://download.pytorch.org/whl/cu121`); plain
  `pip install torch` on Windows is CPU-only. Either way the app works — CPU just embeds
  the images a bit slower. ViT-B/32 (~600 MB) fits easily in the 4 GB VRAM.
- **Why search is fast.** Embedding the images is a **one-time** step — save the vectors
  with `np.save`. Each query then only embeds the short text and does one dot product
  against the saved matrix: milliseconds, even for tens of thousands of images. **Search
  is NumPy on CPU and needs no GPU** (reach for FAISS only at millions of images).
- **Environment = Anaconda3** (Python 3.13 here — new enough for the CUDA wheels). Install
  with `pip install -r requirements.txt`; for GPU install the CUDA torch wheel FIRST
  (see requirements.txt / the notebook's GPU cell), then the rest. A clean env is tidy:
  `conda create -n clip-search python=3.11 -y`.
- **No full download** — use `streaming=True` + `.take(N)` so only the ~1.5k images we
  actually use come over the network (a few hundred MB saved to `images/`), not the
  multi-GB full set. Without streaming, `load_dataset` fetches everything on first call.
